# Quest clear-flag injection — prototype findings

Goal: give the **129 BB quests that record no clear flag** (118 "repeatable", 11
"no-success") a trackable completion flag, so every quest can report player
clears — counted, not just yes/no, since they're repeatable.

Status: **process proven mechanically** on `q237` (MAXIMUM ATTACK 1 Ver2).
End-to-end play-test still pending (see Open questions).

## The injection recipe (verified)

newserv ships both halves of a lossless round-trip. `--reassembly` is the key
flag — the default disassembly is a human listing (address + hex + mnemonic) the
assembler can't read; `--reassembly` emits assembler source with symbolic labels.

```sh
# 1. disassemble to editable assembler source
newserv disassemble-quest-script --bb --reassembly q237-bb-e.bin q237.txt

# 2. insert one line as the FIRST instruction of the set_qt_success target label.
#    q237's success routine is `label015E`; we add `gset 0x02D2` before it:
#      label015E@0x015E:
#        gset                            0x02D2      <-- injected
#        va_start                                    <-- original first instr
#        ...

# 3. reassemble (PRS-compresses by default, like the on-disk format)
newserv assemble-quest-script q237.txt q237-bb-e.bin
```

Verification on q237: assembles to a valid `.bin`, re-disassembles cleanly, gset
count 0 → 1, and `gset 0x02D2` is the first instruction of the success routine.
The **round-trip is exactly lossless** — disassemble→assemble→disassemble on a
quest that already has a gset (q004) produced a byte-identical listing, so the
assembler faithfully relocates everything; inserting one instruction is safe.

## Flag allocation (no collisions)

Quest flags are **0x000–0x3FF** (`QuestFlags = FlagsTable<0x400,4,Difficulty>`;
the 6x75 setter enforces `flag < 0x400`). A scan of all 196 BB quests found
**176 distinct flags written, spanning 0x14–0x3FE**, leaving 848 free — including
a contiguous free run at **0x2D2–0x3FB (298 flags)**.

Allocate injected flags from `0x2D2 + index`, one per unmapped quest (129 needed,
298 available). Completion detection is **scoped to the loaded quest**
(`on_set_quest_flag`: flag set == *this* quest's `completion_flag` while *this*
quest is loaded), so flags could even be reused across quests — but unique
allocation in the clear range is simplest and self-documenting. q237 → `0x02D2`.

## Completion COUNT comes (almost) for free

`on_set_quest_flag` (ReceiveSubcommands.cc:3140) logs a completion on **every**
flag-set where `flag == completion_flag` — it's inside `if (should_set)` with no
0→1 transition gate. So `quest_completions.jsonl` already appends **one record
per clear**. Count tracking is therefore mostly a **dashboard change**: count the
records per (player, quest) and show "cleared N×" instead of a yes/no badge. No
backend change required.

## No-success quests (the harder 11)

Battle (PvP) and a few decoration/lobby quests have no `set_qt_success` routine
at all. Injection there is a bigger edit: register a success routine
(`set_qt_success labelXXXX`) and define `labelXXXX:` with `gset 0xNNNN; ret`,
hooked at the quest's real end point. Lower priority — "completion" is fuzzy for
PvP; tackle after the 118 repeatables.

## Open questions (resolved by play-test)

1. **Does a re-clear re-fire the 6x75?** The server logs per-set, but only if the
   client re-sends the 6x75 when the flag is already set. If it doesn't, a
   persistent flag would count only the first clear. Fallback if so: success
   routine does `gset` then `gclear` so each clear re-transitions.
2. **Does the injected flag fire at all in-game?** Confirm a clear writes
   `Setting quest flag …:02D2` to the log and appends a completions record.

## No-drift architecture (single source of truth)

Every place we record quest data must derive from one authority, never be
hand-kept in parallel:

- **Clear flag** → the per-quest sidecar `q###-bb.json` `CompletionFlag` is the
  source of truth. The injected `.bin` must agree with it. **CI guard:**
  disassemble each quest with a sidecar and fail if its success-routine `gset`
  ≠ the sidecar's `CompletionFlag` (catches .bin/sidecar drift).
- **Injected `.bin`s** should be *generated* from the image's vanilla quest +
  the flag spec, so a future vanilla-quest update re-injects cleanly rather than
  leaving a stale hand-edited binary.
- **Markdown catalog** (`docs/quest-catalog.md`) → generated by a committed
  script from live `/api/quests` + sidecars + provenance; **CI guard:**
  regenerate and fail if it differs from the committed copy.
- **Dashboard** → reads the same live API + sidecars, so it can't drift.

## Picking a level-appropriate test quest

A quest's actual areas aren't in the script (they're set via register-driven
`map_designate`) but ARE in its map: `disassemble-quest-map --bb q###-bb.dat`
lists every entity with a `floor=` (0 = Pioneer 2, 1-2 Forest, 3-5 Caves, 6-7
Mines, 8-10 Ruins, 11-14 bosses). Scanning the Episode-1 repeatables this way
picked **q101 "Mop-up Operation #1"** for a level-3 character: Pioneer 2 +
Forest 1 only, no boss, `no_gset`. (q237 MAXIMUM ATTACK 1 is a long horde and
q58 Lost HEAT SWORD ends at the Dragon — both rejected.)
